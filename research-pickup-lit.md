## 一句话结论速览（给团队主理人）

- **体裁一句话**：捡手机文学 = 以"捡到别人手机"为叙事框架、用高度拟物的虚构微信聊天记录（私聊/群聊）展开故事的沉浸式网络文学，靠对话推剧情、靠留白造悬疑。

## 一、捡手机文学体裁研究

### 1.1 定义与起源

**定义**：捡手机文学（found-phone / phone-dump 体）是一种以"我捡到了一部手机（或登错别人账号）"为叙事框架，通过**展示这部"捡来"的手机里的聊天记录、朋友圈、备忘录等内容**来虚构故事的网络文学形式。读者以"第三人称窥探者"视角阅读，获得"像真的捡到别人手机"的偷窥感与沉浸感。

**起源脉络**（据公开资料交叉印证）：
- **2016 年前后**：日韩、欧美先出现以聊天气泡对话推动情节的"对话体小说（chat fiction）"。
- **国内起点**：最早源自**豆瓣韩式泡菜小组**的饭圈同人创作——粉丝代入真人明星名字、用 PS 捏造微信聊天记录，标题多为"今天捡到了 XX 的手机…"，以自娱自乐为主。
- **破圈**：逐渐脱离偶像题材，发展为全民可玩的原创故事形式，并衍生出"论坛盖楼体 / 知乎体 / 弹幕体 / 直播体"等变体。
- **流量峰值**：抖音话题 `#捡手机文学` 播放量达 **55.6 亿次**；在小红书、微博、LOFTER 广泛传播。

> ⚠️ **法律风险提示（应写进产品默认免责/引导）**：早期大量作品冒用真人姓名编故事，曾引发名誉权侵权判例（如 2021 杭州取快递被造谣事件）。正规创作已约定俗成使用**原创虚拟角色** + 文末"ooc 致歉 / 请勿上升真人"免责声明。工具应在新建项目时引导使用虚拟角色。

### 1.2 核心特征与叙事手法

| 特征 | 说明 | 对工具的启示 |
|---|---|---|
| **高度拟物（拟真 UI）** | 精细模仿手机系统：信号格、电量、时间戳、输入框、甚至"对方正在输入…" | 预览必须"像微信"，细节越真越有说服力 |
| **对话即剧情** | 全篇无大段旁白/心理描写，角色脾气、矛盾、感情拉扯全藏在一句句聊天里 | 编辑核心是"气泡"，不是"段落" |
| **第三人称窥探视角** | 读者是"捡手机的人"，不是对话当事人 → 安全偷窥体验 | 叙事框架（捡手机开场）可作为可选模板 |
| **私聊 + 群聊 + 朋友圈** | 单聊走亲密/悬疑，群聊走群像/热闹；朋友圈碎碎念补人物 | 必须同时支持单聊与群聊，且要"作为谁说" |
| **微表情细节** | 表情包、消息撤回、拍一拍、"对方正在输入"等 | 撤回/拍一拍/引用/拉黑是"卖点级"模拟功能 |
| **留白与悬念** | 常在某条消息后戛然而止，诱导读者猜后续、去评论区催更 | 工具应支持"断在哪一刻"，长图导出便于分享 |

**情绪/悬疑/日常的营造方式**：
- **悬疑**：用"半截对话 + 未读红点 + 时间跳跃"制造信息差；读者自己脑补空白。
- **甜/日常**：用互怼→暧昧的对话节奏、表情包、撤回又重发的小动作。
- **破防/现实**：用群聊里众人的沉默、某条系统提示（"消息已发出，但被对方拒收了"）。

### 1.3 典型结构

- **固定开篇**：①"在地铁上捡到一个手机，里面的聊天记录让我…"；②"不小心登错了别人的微信"。
- **单聊 vs 群聊**：
  - 单聊：两人对话，节奏紧凑，适合甜文/悬疑/情感拉扯。
  - 群聊：3+ 人，群像戏、吃瓜、热搜式话题，适合"班级群/家族群/公司群"等设定。
- **时间线**：多为线性（按消息时间），也常见"几天后""凌晨 3:14"式时间跳切制造张力。
- **收尾 / 留白**：常在关键消息后停止（"对方已退出聊天" / 红点未读 / 最后一条语音），把情绪留给读者；或用一个朋友圈/系统提示收束。

### 1.4 为什么在社交平台受欢迎

1. **窥私欲的合规满足**：提供"安全偷窥他人生活"的体验，是当代社交时代的普遍心理。
2. **接地气 + 低门槛**：校园甜恋、职场吐槽、家庭日常，题材贴近生活；篇幅短、对话体，通勤/摸鱼随手刷完。
3. **强代入与情感共鸣**："原来不只我这样"的安慰感；部分人获得虚拟社交满足。
4. **互动性 / 催更文化**：作者末尾留"看完请留评""蹲后续"，形成评论区追踪-反馈短闭环。
5. **技术驱动降门槛**：Talkmaker、拟物生成器等"聊天记录生成器"App 让素人一键生成截图，催生大量 UGC。
6. **算法友好**：截图式内容在小红书/抖音信息流里天然吸睛、易转发。

### 1.5 作者常见痛点（→ 反推我们工具的卖点）

现有主流工具是 **PS** 或 **聊天记录生成器 App（Talkmaker 等）**，它们是"截图生成器"，不是"写作/编辑工具"。痛点即卖点：

| 痛点 | 现有工具现状 | 我们的卖点（反推） |
|---|---|---|
| **改一个字要重做** | 生成器截图后无法回到气泡级编辑 | 气泡级"方便编辑/删除/重排"，作者模式随时改 |
| **切换"作为谁说"繁琐** | 多数生成器仅支持左右两角色，群聊切换麻烦 | 单聊一键切换；群聊成员选择器"作为谁说" |
| **群聊成员管理乱** | 加人/改名/换头像要反复操作 | 聊天信息页集中管理成员（昵称+头像+isMe） |
| **图片只能上传，不能粘贴** | 桌面想贴微信截图很麻烦 | 桌面**剪切板粘贴**图片（复用现有 `makePasteHandler`） |
| **语音条/转文字缺失** | 生成器基本不支持语音条 | 支持语音条（时长 + 下方转文字，作者可编辑） |
| **模拟动作手动伪造** | 撤回/引用/拍一拍/拉黑要自己 PS | 内置"撤回/引用/拍一拍/拉黑"一键模拟，风格统一 |
| **多部作品散乱** | 截图散落相册、多个 App 草稿 | **统一项目管理页**，多部捡手机作品集中管理 |
| **导出拼图麻烦** | 需另用拼图 App 裁切长图 | 沿用现有 **长图导出（canvas）**，直接出微信式长图 |
| **多端不一致** | 手机生成器 ≠ 电脑精细编辑 | 单文件 + 响应式，**手机写、电脑精修**同一份数据 |
| **数据易丢** | 生成器云端/本地易失 | **IndexedDB 本地持久化**，零依赖离线可用 |

## 二、聊天 UI 范式对比（微信 / iMessage / KakaoTalk）

| 维度 | 微信 WeChat | iMessage (iOS) | KakaoTalk (韩) |
|---|---|---|---|
| **气泡布局** | 左右分栏：自己右（绿）、对方左（白） | 左右分栏：自己右（蓝）、对方左（灰），偏居中留白 | 左右分栏：自己右（黄/灰）、对方左 |
| **群聊支持** | ✅ 完善，群内显示成员**昵称** | ✅ 支持，但自己永远在右，无"作为他人说"概念 | ✅ 支持，群内显示昵称 |
| **昵称/头像展示** | 对方气泡上方显示昵称+头像；群聊每条都带昵称 | 群聊顶部显示发信人小字；单聊不显示昵称 | 群聊显示昵称+头像，风格圆润可爱 |
| **语音条样式** | 右侧▶按钮 + 波形条 + **时长（秒）**，下方可转文字 | 波形条 + 时长，无"转文字"原生习惯 | 类似，时长显示 |
| **引用** | ✅ 长按引用，气泡顶部显示引用块（灰底） | ✅ 回复带引用缩略 | ✅ 引用回复 |
| **撤回提示** | 居中灰字"XXX 撤回了一条消息" | 居中提示"你撤回了一条信息" | 居中提示"OOO 撤回了消息" |
| **拍一拍 / 表情** | ✅ 双击头像"你拍了拍 XXX"；emoji+微信表情 | ❌ 无拍一拍；富气泡效果 | ✅ 有"DDK"等；大表情动画丰富 |
| **拉黑/拒收提示** | ✅ "消息已发出，但被对方拒收了"（强叙事张力） | ❌ 无对应语义 | 弱 |
| **整体视觉重量** | 中（绿白、克制） | 轻（蓝白、留白多） | 偏重（明黄、圆润、表情重） |
| **国内读者认知度** | ⭐⭐⭐⭐⭐ 默认心智模型 | ⭐⭐ 部分年轻人知道 | ⭐ 基本无 |
| **极简 + 作者可编辑契合度** | 高：结构规整、状态行统一、易程序化 | 中：自我-右模型与"作为谁说"冲突 | 中：圆润风格难"极简" |

### 推荐结论：**采用微信风格**

**理由（基于功能契合度论证，与用户倾向一致）：**

1. **读者认知锚定微信**：体裁 99% 以微信聊天记录呈现（抖音 55.6 亿播放、Talkmaker 等工具默认即微信 UI）。用微信风格 = 读者零学习成本、最强"以假乱真"沉浸感；iMessage/KakaoTalk 在国内读者心智中陌生，会立刻"出戏"。
2. **需求字段被微信原生覆盖**：左右气泡、群聊昵称、引用块、撤回提示、拍一拍、拉黑拒收提示——本需求第 4/5/8/11 条全部能在微信范式下"零改造"映射。iMessage 无拍一拍/拉黑语义，KakaoTalk 国内读者无感。
3. **"作为谁说"与微信模型天然兼容**：微信中每条消息归属明确发送者（含群聊昵称），作者只需切换"当前发送者身份"即可把气泡放到左或右——而 iMessage 的"自己永远在右侧"模型与"我要以对方身份发言"直接冲突，实现别扭。
4. **极简且易程序化**：微信 UI 结构规整（状态行/系统提示统一居中灰字），便于我们复用现有 `phone-frame` + canvas 长图渲染管线，作者模式下也最易加 hover/长按编辑入口。
5. **情感张力道具齐全**：拉黑后的"消息已发出，但被对方拒收了"、拍一拍、撤回——正是捡手机文学制造破防/甜点的关键微表情，微信提供现成范式。

> 备注：若未来想做"国际化/不同手机品牌"变体（如模拟 iOS 深色模式、华为/小米主题），可在 `settings.theme` 预留扩展，但**首版只做微信一套**，避免分散。

## 三、产品信息架构与交互方案（在现有单文件应用内扩展）

### 3.1 三层导航

```
首页（项目管理页）            ← 现有 renderHome()，两类项目同屏
  └─ 某「捡手机」项目          ← 新增：类微信的「聊天列表」页
        └─ 某条聊天            ← 新增：类微信的「聊天详情」页（气泡流 + 底部输入栏）
```

- **首页**：现有卡片网格直接复用，卡片显示标题 + "N 个聊天 · 更新时间"；「新建项目」按钮改为**先选类型**（论坛体 / 捡手机）的弹窗（见 3.3）。论坛体与捡手机卡片同屏，满足需求 1。
- **项目内（捡手机）**：进入后是**聊天列表**（仿微信会话列表），可上滑浏览、可增/删聊天（需求 6）。
- **聊天详情**：微信式气泡流，底部输入栏（需求 3/4/5/7/8/9/10/11）。

### 3.2 响应式布局建议（手机 / 平板 / 桌面）

**手机端（首屏 = 项目管理页）**：
```
┌─────────────────────┐
│  ← 我的作品          ＋ │   ← 项目管理页（卡片/列表）
│ ┌────┐ ┌────┐       │
│ │小说A│ │捡手机B│     │   ← 两类同屏
│ └────┘ └────┘       │
└─────────────────────┘
       点击「捡手机B」→
┌─────────────────────┐
│  ← 捡手机B     ⓘ     │   ← 聊天列表（微信会话列表）
│ 💬 班级群       21:03 │
│ 💬 他           昨天  │
│ ＋ 新建聊天             │
└─────────────────────┘
       点击某聊天 →
┌─────────────────────┐
│  ← 班级群        ⓘ   │   ← 聊天详情（全屏，微信式）
│  ┌──┐ 在吗？          │
│ 嗯📎  ├──┐           │
│      │图片│           │
│ [引用: …] 我到了       │
│ ┌──────────────────┐ │
│ │[👤 我] 说点什么… 🖼️│ │   ← 输入框左侧=发言身份按钮
│ └──────────────────┘ │
└─────────────────────┘
```

**平板端**：聊天列表与详情可左右分栏（仿微信平板），列表窄栏 + 详情主区；增删聊天/成员在详情页内完成。

**桌面端（推荐三栏，而非沿用现有"编辑/预览"双栏）**：
```
┌──────────┬──────────────────────┬──────────────────┐
│ 聊天列表  │   聊天详情（微信式预览+编辑合一）│  作者属性面板（右栏） │
│ 班级群    │   （气泡流，作者模式可hover编辑） │  当前发言身份：[我▾]   │
│ 他        │                        │  成员管理：[＋ 添加]   │
│ ＋新建     │   [👤我] 说点什么… 🖼️│  张三 ✎ 🗑         │
│          │                        │  李四 ✎ 🗑         │
│          │                        │  语音：模拟▾ / 录制  │
│          │                        │  该消息：引用 撤回 拍一拍 拉黑│
└──────────┴──────────────────────┴──────────────────┘
```
> 桌面用**三栏**：左=聊天列表，中=聊天详情（微信式，编辑即所见），右=作者属性/操作面板（切身份、管成员、编辑转文字、设撤回/拍一拍/拉黑）。右栏在手机/平板收起为**抽屉/弹窗**（点 ⓘ 或身份按钮呼出），保证"界面极尽简单"。

### 3.3 各功能交互方案与草图（文字版）

**① 发言人切换按钮（输入框左侧）**
- 单聊：左侧按钮显示当前身份头像+名（默认"我"），点击在「我 ↔ 对面」间切换；气泡随之落到右/左。
- 群聊：左侧按钮显示「作为：XXX ▾」，点击展开**成员列表选择器**（"作为谁说"），点选后后续消息以该成员身份发出。
- 草图：`[👤我 ▾] 说点什么… 🖼️` ；群聊展开为浮层成员列表（带头像/昵称/「设为我」）。

**② 群聊成员管理与"作为谁说"选择器**
- 入口：聊天详情右上 `ⓘ` → 聊天信息页。
- 功能：增/删成员；每个成员设**昵称 + 头像**（头像复用现有 `cover-circle` 上传/粘贴逻辑）；指定一个成员为 `isMe`（"我"）。
- ">2 人即群聊"由成员数自动判定：成员数=2 → 单聊（左右两角色）；≥3 → 群聊（每条消息带昵称）。
- 发送时从 ① 的按钮选择"作为谁说"。

**③ 语音消息：真实录音 vs 模拟 + 转文字编辑**
- **推荐：模拟优先，真实录制可选。**
  - 模拟：作者填「时长（秒）」+ 在气泡下方写「转文字内容」；渲染为微信式语音条（▶ + 波形 + 时长），条下方显示转文字，作者可点击**行内编辑**（textarea 浮层）。
  - 真实录制：零依赖可用 `MediaRecorder` + `getUserMedia` 录成音频（存 dataURL/blob），再手填转文字（离线无 ASR，故转文字仍需作者写；可后续接在线 ASR 作为增强）。
- 理由：体裁核心卖点是"语音条样子 + 转文字文本"，真实录音非必需，先模拟即可 100% 覆盖需求 8。

**④ 气泡的编辑 / 删除入口（作者模式）**
- 进入捡手机项目默认开启「作者模式」（顶部可切"读者预览"）。
- 桌面：鼠标 **hover 气泡** → 右上浮出 `✎ 编辑 / 🗑 删除 / ↩ 撤回 / 🔗 引用 / 👋 拍一拍` 小工具条。
- 移动端：**长按气泡** → 底部 ActionSheet 同上。
- 删除即移除该气泡；编辑即就地改文本/图片/转文字，实时刷新微信式预览。满足需求 9。

**⑤ 图片上传 + 桌面剪切板粘贴**
- 输入框旁 `🖼️` 按钮：触发文件选择（移动端/桌面通用，**上传**）。
- 桌面端：输入框/聊天区监听 `paste` → 调用现有 `makePasteHandler` 把剪切板图片转为待发送图片（需求 10）。移动端因系统限制仅支持上传（在 UI 上不显示粘贴提示，避免误导）。
- 图片存为 `{id, dataUrl, name}`，与现有论坛体图片结构一致，便于复用长图导出。`makePasteHandler` / `handleImageFiles` 可直接复用。

**⑥ 引用 / 撤回 / 拉黑 / 拍一拍 的呈现与模拟逻辑**
- **引用**：长按某气泡选「引用」→ 输入框上方出现引用块（灰底 + 原文片段）→ 发送后新气泡顶部显示该引用块。
- **撤回**：作者对某气泡选「撤回」→ 该消息标记 `withdrawn=true`，渲染为**居中灰字系统提示"XXX 撤回了一条消息"**；作者模式下可「恢复」（取消 withdrawn，气泡重现）。
- **拉黑**：在成员/聊天设置中标记某成员 `blocked`（关系态）→ 从该成员发出的、处于"我"视角的后续消息渲染为**系统提示"消息已发出，但被对方拒收了"**（或拦截提示条）。满足"拉黑"叙事张力。
- **拍一拍**：作者对成员触发拍一拍 → 插入**系统提示"你拍了拍 XXX"**（若以对方面视角则为"XXX 拍了拍 YYY"）。双击头像即可触发，仿微信。

### 3.4 微信气泡细节补充（最新设计反馈）

**⑨ 语音消息开关**
- 输入栏右侧、图片按钮旁边设置一个「语音」开关按钮（默认关）。
- **关闭状态**：输入框打字 → 发送普通文字气泡。
- **开启状态**：输入框打字变成「语音消息的转文字内容」；点击发送时先弹出一个小浮层/弹窗让作者填写**语音秒数**（如 3），确认后插入一条语音气泡（语音条 + 转文字）。
- **发完自动复位**：语音消息发出后开关自动恢复为关闭，避免下一条误发成语音；作者也可手动随时切换。
- 语音气泡样式：绿色（我）/白色（对方）的细长圆角矩形，左侧▶按钮 + 模拟波形条 + 右侧显示秒数，**下方居中显示转文字**（可再次编辑）。
- 手机端同样使用该开关；真实录音（`MediaRecorder`）作为后续增强，首版只模拟。

**⑩ 头像、连续对齐、气泡小三角**
- **每条消息都显示头像**：单聊对方头像在左、"我"头像在右；群聊每条消息上方显示昵称，头像在气泡外侧。不允许连续消息只保留第一条头像。
- **同一发送方连续多发**：所有气泡统一沿同一方向完整对齐（右侧全靠右 / 左侧全靠左），每条气泡独立保留头像；气泡之间用紧凑间距（4–6px）表达连续性，而不是越靠越近。
- **气泡外侧加小三角指向说话人**：微信左侧白气泡在**左边缘**伸出同色系小三角指向左方头像；右侧绿气泡在**右边缘**伸出同色系小三角指向右方头像，三角与气泡连成一体。
- 这些细节保证预览"像微信"，也是长图导出时读者沉浸感的来源。

## 四、数据模型草案（JSON 结构示意）

### 4.1 与现有论坛体 Project 的共存方案

现有 `Project`：`{ id, title, createdAt, updatedAt, data }`，`data` 为论坛体专用。
**方案：在 Project 顶层新增 `type` 字段区分类型；`data` 内容随 `type` 变化。** 旧论坛体项目无 `type` → `normalizeProject` 默认补 `'forum'`，**100% 向后兼容**。

```jsonc
// 统一外层（两种类型共用，IndexedDB store 'projects' 不变）
{
  "id": "uuid",
  "title": "未命名作品",
  "createdAt": 1700000000000,
  "updatedAt": 1700000000000,
  "type": "forum" | "pickup",   // 新增：缺省视为 'forum'（兼容旧数据）
  "data": { }                    // 随 type 不同而不同（见下）
}
```

首页 `renderHome` 按 `type` 渲染不同卡片副标题（"N 层" / "N 个聊天"）；「新建项目」弹窗二选一；`openProject` 按 `type` 路由到不同编辑器（论坛体沿用现逻辑，捡手机走新增逻辑）。`exportJSON` 已含 `version` + `project`，天然兼容。

### 4.2 捡手机项目 data 草案

```jsonc
{
  "type": "pickup",
  "data": {
    "settings": {
      "showTime": true,          // 是否显示每条消息时间戳
      "theme": "wechat",         // 预留：wechat / future
      "cleanExport": false       // 纯净导出：true=隐藏撤回/拒收等系统提示
    },
    "chats": [ /* Chat */ ]
  }
}
```

```jsonc
// Chat：一个会话（仿微信一个联系人/群）
{
  "id": "uuid",
  "name": "班级群",              // 会话标题
  "type": "single" | "group",   // 由成员数自动判定：2=single，>=3=group
  "order": 0,                    // 聊天列表排序
  "members": [ /* Member */ ],
  "messages": [ /* Message */ ]
}
```

```jsonc
// Member：会话中的角色
{
  "id": "uuid",
  "name": "张三",               // 昵称
  "avatar": { "id":"uuid", "dataUrl":"data:image/...", "name":"a.png" } | null,
  "isMe": true,                 // 谁是"我"（右侧默认发言者）
  "blocked": false              // 拉黑关系态：true=该成员消息在"我"视角显示拒收提示
}
```

```jsonc
// Message：一条消息/系统提示
{
  "id": "uuid",
  "senderId": "member-uuid",    // 发送者（Member.id）；系统提示可填触发者
  "type": "text" | "image" | "voice",
  "text": "在吗？",             // text 正文；image 时为可选配文；voice 时为空
  "image": { "id":"uuid", "dataUrl":"...", "name":"x.png" } | null,
  "voice": {                    // type==='voice' 时
    "durationSec": 3,           // 语音条显示多少秒
    "transcript": "你到家了吗"   // 下方转文字，作者可编辑
  } | null,
  "quote": { "messageId":"uuid", "snippet":"原文前 N 字" } | null,  // 引用
  "withdrawn": false,           // true => 显示"XXX 撤回了一条消息"，原内容隐藏可恢复
  "nudge": { "fromId":"uuid", "toId":"uuid" } | null,  // 拍一拍 => "你拍了拍 XXX"
  "isSystem": false,            // true=纯系统行（撤回/拍一拍/拒收本身作为系统提示）
  "time": 1700000123000         // 时间戳
}
```

**字段语义小结（对应需求）：**
- `type: text|image|voice` + `voice.transcript` → 需求 8（语音条 + 转文字可编辑）
- `senderId` + `members[].isMe` → 需求 4/5/7（切换"是谁在说"、我是谁/对面是谁）
- `members` 数组长度 → 需求 7（>2 人即群聊）
- `quote` → 需求 11（引用）
- `withdrawn` → 需求 11（撤回）
- `members[].blocked` + 渲染层系统提示 → 需求 11（拉黑）
- `nudge` → 需求 11（拍一拍）
- `image` 支持 `makePasteHandler` 粘贴 → 需求 10
- 图片/头像均用现有 `{id, dataUrl, name}` 结构 → 与论坛体一致，长图导出管线可复用

## 五、待与用户确认的关键决策清单

> 每条给出**推荐选项**，便于主理人直接拿去和用户讨论。

| # | 决策点 | 推荐选项 | 备选 / 说明 |
|---|---|---|---|
| 1 | **聊天 UI 风格** | 微信（理由见第二节） | iMessage / KakaoTalk（国内读者易出戏，不推荐） |
| 2 | **语音消息：真实录制 vs 模拟** | 模拟优先（填时长+转文字），真实录制（`MediaRecorder`）作为可选 | 纯模拟最省事；真实录制增加离线音频存储体积 |
| 3 | **撤回 / 拉黑 / 拒收 等模拟动作，在「成品导出（长图/HTML）」中是否保留** | 默认**呈现为系统提示**（保沉浸），提供 `cleanExport` 开关导出纯净版 | 若全隐藏则丢失体裁张力 |
| 4 | **桌面端布局** | **三栏**：聊天列表｜聊天详情｜作者属性面板（右栏手机端收为抽屉） | 沿用现有"编辑/预览"双栏（但聊天体更需要"所见即微信"，三栏更贴合） |
| 5 | **作者模式下气泡编辑/删除的触发** | 桌面 **hover 浮层** + 移动端 **长按 ActionSheet** | 也可统一用长按，但桌面 hover 更高效 |
| 6 | **「新建项目」如何区分两种类型** | 首页「＋新建」弹窗**二选一**（论坛体 / 捡手机） | 或首页分类标签+各自新建按钮 |
| 7 | **图片体积与存储** | 上传/粘贴时**按比例压缩**（如长边≤1280）再存 dataURL；导出长图时二次压缩 | 单文件+IndexedDB 对多图长聊天有体积膨胀风险，需压缩兜底 |
| 8 | **导出格式期望** | 微信式**长图**（沿用 canvas 管线）+ **可编辑 HTML** + **JSON**；支持「整本 / 按单个聊天」导出 | 是否需要 PDF？是否要" consecutive 长图拼接"？ |
| 9 | **拟物细节「对方正在输入…/已读/红点未读」是否做** | 作为**可选模拟元素**，默认关；开场模板可带"捡到手机"框架 | 过度拟物会增加编辑复杂度，建议克制 |
| 10 | **多项目数据隔离与备份** | 沿用 IndexedDB；导出 JSON 即备份；后续可加「导入」复用现有 `triggerImport` | 是否需云端同步？（本轮不做，保持零依赖） |

**我的首推拍板顺序（建议先和用户确认）**：① UI=微信（已倾向） → ② 语音模拟优先 → ③ 撤回/拉黑导出呈现方式 → ④ 桌面三栏布局。这四项确定后，其余可在开工后边做边定。
